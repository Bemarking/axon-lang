# Database Foundation — Operator Guide

Axon Enterprise v1.1.0+ runs on **PostgreSQL 16+** with Row-Level
Security policies enforced on every tenant-scoped table. The schema is
split between two owners:

| Schema | Owner | Contents |
|---|---|---|
| `public` | axon-lang Rust runtime | `tenants`, `traces`, `flows`, operational data |
| `axon_control` | axon-enterprise Python | users, roles, audit events, metering, secrets metadata |

Both sides share the same database connection and the same RLS GUC
(`axon.current_tenant`) — isolation is enforced by Postgres, not by
application code.

## Required roles

| Role | Login | Flags | Used by |
|---|---|---|---|
| `axon_migrator` | yes | `CREATEROLE`, `CREATE` on target DB | Alembic runner / ops team only |
| `axon_admin` | no | `BYPASSRLS` | Admin API, migrations, audit verification |
| `axon_app` | yes | none (RLS applies) | application workers / HTTP handlers |

The baseline migration creates `axon_admin`. The other two roles are
provisioned by operators out-of-band (they need cluster-level privileges
that Alembic refuses to request on its own).

### Bootstrapping the roles (one-time)

```sql
-- As a superuser on a fresh RDS instance:

CREATE ROLE axon_migrator LOGIN PASSWORD :'migrator_pw' CREATEROLE;
GRANT CREATE ON DATABASE axon_enterprise TO axon_migrator;

CREATE ROLE axon_app LOGIN PASSWORD :'app_pw';
GRANT CONNECT ON DATABASE axon_enterprise TO axon_app;

-- axon_admin is created by the baseline migration; grant it to axon_app
-- ONLY for admin-plane operations (the Admin API connects as a member of
-- axon_admin; tenant-plane handlers connect plain as axon_app).
```

## Running migrations

Migrations are versioned in `alembic/versions/` and run via the
standard Alembic commands. The URL is resolved from `AXON_DB_URL`
(the same environment variable the app uses) so there is exactly one
source of truth.

```bash
# Upgrade to head (production path — inside the deploy pipeline):
export AXON_DB_URL="postgresql+asyncpg://axon_migrator:...@db:5432/axon_enterprise"
alembic upgrade head

# Preview the SQL without applying (code review):
alembic upgrade head --sql > pending.sql

# Downgrade one revision (rollback):
alembic downgrade -1

# Generate a new revision against the current model set:
alembic revision --autogenerate -m "add_user_table"
```

Autogenerated revisions are **always** hand-reviewed — the template
runs `black` and `ruff --fix` as post-write hooks so diffs stay
focused on intent, not formatting.

## Row-Level Security model

Every tenant-scoped table carries:

- `tenant_id TEXT NOT NULL REFERENCES axon_admin.tenants(tenant_id)`
- RLS enabled **and forced** (`FORCE ROW LEVEL SECURITY`)
- Two policies emitted by `rls_policies.full_policy_set_sql()`:
  - `tenant_isolation` — `tenant_id = current_setting('axon.current_tenant', true)`
  - `admin_bypass` — `TO axon_admin USING (true)`

`FORCE` is important: without it, the table owner (the connecting
role) would bypass the policy, which defeats the whole model. The
migration pattern is therefore:

```python
from axon_enterprise.db.rls_policies import full_policy_set_sql

def upgrade() -> None:
    op.create_table("users", ...)
    for stmt in full_policy_set_sql(table="users"):
        op.execute(stmt)
```

## Sessions from application code

Three context managers, each a single transaction:

```python
from axon_enterprise.db.session import tenant_session, admin_session, read_session

# Tenant-scoped read/write (explicit context):
async with tenant_session(ctx) as session:
    session.add(Widget(tenant_id=ctx.tenant_id, ...))

# Tenant-scoped (implicit — reads from ContextVar set by middleware):
async with tenant_session() as session:
    ...

# Cross-tenant admin, connects as axon_admin, emits audit events:
async with admin_session() as session:
    rows = await session.execute(select(Widget))   # BYPASSRLS

# Read-only, routes to replica when configured:
async with read_session(ctx) as session:
    ...
```

Transaction semantics are deterministic:

- **Commit** on clean exit.
- **Rollback** on exception.
- `SET LOCAL axon.current_tenant = :id` scopes the GUC to the tx, so
  a pool-recycled connection never carries a stale tenant into the
  next request.

## Connection pool tuning

Defaults target a single 1 GB service container. Scale horizontally by
increasing container count, not pool size — more than ~30 connections
per pod contends with Postgres' own connection limits on `db.t3.*`
instances.

| Setting | Default | Override |
|---|---|---|
| `pool_size` | 10 | `AXON_DB__POOL_SIZE` |
| `max_overflow` | 20 | `AXON_DB__MAX_OVERFLOW` |
| `pool_timeout_seconds` | 30 | `AXON_DB__POOL_TIMEOUT_SECONDS` |
| `pool_recycle_seconds` | 1800 | `AXON_DB__POOL_RECYCLE_SECONDS` |
| `statement_timeout_ms` | 30000 | `AXON_DB__STATEMENT_TIMEOUT_MS` |
| `idle_in_transaction_timeout_ms` | 60000 | `AXON_DB__IDLE_IN_TRANSACTION_TIMEOUT_MS` |
| `lock_timeout_ms` | 5000 | `AXON_DB__LOCK_TIMEOUT_MS` |
| `slow_query_ms` | 1000 | `AXON_DB__SLOW_QUERY_MS` |

Slow queries are logged as structured events under the
`axon_enterprise.db.engine` logger with a `slow_query` event name.
Sampling to a metrics backend happens in Fase 10.i.

## Testing

Integration tests spin up an ephemeral Postgres 16 via testcontainers.
The session-scoped fixture applies a minimal `axon_admin.tenants` seed
(standing in for the Rust M1 migration) and runs `alembic upgrade head`
before yielding a live engine. Tests are selected with:

```bash
pytest -m integration                 # run only integration tests
pytest -m "not integration"           # run only pure-Python tests
pytest                                # run both (default)
```

Docker is required for integration; the suite auto-skips when the
daemon is unreachable so developer machines without Docker still get
green on pure-Python changes.
