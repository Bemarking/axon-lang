# ‚ñ∂ 1. The kivi smoke-16 finding #1 (2026-05-20)

> *"IDENTITY no surfado por axon-server store introspect + T803 dispara
> est√°ticamente ‚Äî kivi tuvo que omitir `chat_history.id` del schema.

The adopter's manifest looked like (paraphrased):

```json
{
  "version": 1,
  "stores": {
    "public.chat_history": {
      "columns": {
        "id": { "type": "BigInt", "primary_key": true, "not_null": true },
        "tenant_id": { "type": "Uuid", "not_null": true },
        "content": { "type": "Text", "not_null": true },
        ...
      }
    }
  }
}
```

with the live DB shape:

```sql
CREATE TABLE chat_history (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    tenant_id   UUID NOT NULL,
    content     TEXT NOT NULL,
    ...
);
```

When the adopter writes `persist into chat_history { tenant_id: ${tid}, content: ${msg} }` (omitting `id`, because Postgres auto-fills IDENTITY), the type-checker fires `axon-T803`:

> `axon-T803 persist omits NOT-NULL column \`id\` (BigInt, primary_key) with no default; the column must be provided or have a default in the schema declaration.`

The workaround the adopter had to apply: **delete `id` from the
manifest entirely**. That makes the persist compile, but it ALSO
disables every other proof for the column ‚Äî a `where: "id = ${target_id}"`
that misnames the column will pass T801 silently because the column
doesn't exist in the type-checker's worldview. **The whole point of
Fase 38 (declared schema ‚Üí compile-time proof) is undermined**.

# ‚ñ∂ 2. Root cause: introspect's two-channel default detection is single-channel

Postgres represents auto-fill defaults via TWO independent channels:

1. **`pg_attrdef.adbin`** ‚Äî captures the textual default expression
   for columns declared with `DEFAULT <expr>`. The legacy SERIAL
   pattern (`column SERIAL` ‚Üí `column INT DEFAULT nextval(seq)`)
   lives here.
2. **`pg_attribute.attidentity`** ‚Äî a single-char marker:
   - `'a'` = `GENERATED ALWAYS AS IDENTITY` (Postgres rejects user-supplied values)
   - `'d'` = `GENERATED BY DEFAULT AS IDENTITY` (Postgres auto-fills when omitted)
   - empty / NULL = not an identity column

axon's introspect (`axon-rs/src/store/introspect_cli.rs:222-260`) reads
channel #1 only. The `detect_auto_increment(default_expression)` helper
in `axon-frontend/src/store_introspect.rs:117` looks for `nextval(`
substring ‚Äî perfect for SERIAL, blind to IDENTITY.

The comment block at `store_introspect.rs:108-114` even acknowledges
the gap:

> *"A column declared `GENERATED ... AS IDENTITY` does NOT appear in
> `pg_attrdef` (Postgres handles identity separately), and
> adopter-authored sequences via `DEFAULT nextval(...)` round-trip here too."*

The bug is the language code BENEATH that doc-comment that should have
queried `attidentity` but didn't. v1.38.3 closes it.

# ‚ñ∂ 3. The IDENTITY Recognition Contract (the heart)

For every column the introspect probe observes:

**D1 ‚Äî `attidentity` is a first-class introspection field.** The deep
`pg_catalog` query selects `a.attidentity AS identity_kind` alongside
the existing `attnotnull` / `pg_get_expr(adbin)` / primary-key / unique
attributes. The Rust-side decoder maps the returned text to
`identity_kind: Option<char>` (Some('a') / Some('d') / None).

**D2 ‚Äî `identity: bool` is a first-class manifest + AST field.**
`StoreColumn` (compile-time AST), `ManifestColumn` (off-line manifest
storage), and `DeclaredColumn` (type-checker's resolved view) all gain
a `pub identity: bool` field. The field is OPTIONAL on read (`#[serde(default)]`)
and SKIPPED on write when false (`#[serde(skip_serializing_if = is_false)]`)
so manifests pre-v1.38.3 round-trip byte-identically.

**D3 ‚Äî T803 treats `identity` as "safely omittable".** The `has_default`
derivation in `DeclaredColumn::from_inline_columns` and `from_manifest_store`
extends to:
```
has_default = !default_value.is_empty() || auto_increment || identity
```
A NOT-NULL identity column is now structurally exempt from the T803
NOT-NULL-omission check.

**D4 ‚Äî `auto_increment` semantics unchanged.** `detect_auto_increment`
still returns true iff the default expression contains `nextval(`. The
adopter who hand-writes a manifest can declare EITHER `auto_increment: true`
(legacy SERIAL) OR `identity: true` (IDENTITY syntax) ‚Äî they're
distinct DDL surfaces and the manifest preserves the distinction.
Introspect emits `identity: true` when `attidentity != ''`; for an
IDENTITY column it does NOT set `auto_increment: true` (the column has
no `nextval` default expression). For a SERIAL column it sets
`auto_increment: true` (the nextval default is observable) and leaves
`identity: false`.

**D5 ‚Äî Absolute backwards-compat.** A manifest written against v1.38.2
parses byte-identically against v1.38.3. The new field's `Default::default()`
is `false`, matching the v1.38.2 semantics for every existing column.
An adopter who upgrades from v1.38.2 to v1.38.3 sees:
- new `axon store introspect` output that includes `identity: true` on
  identity columns
- T803 stops firing on those columns
- nothing else changes

# ‚ñ∂ 5. Forward-compatibility commitments

- `identity` is the SQL-level marker for the AUTO-FILL semantics; the
  manifest field is named after the DDL keyword so adopter docs map
  1:1 to the manifest. If Postgres adds another auto-fill mechanism in
  a future major (e.g. a hypothetical "auto-generated UUID column"),
  the manifest can grow a new optional field without re-purposing
  `identity`.
- `IntrospectionRow.identity_kind` exposes `'a'` vs `'d'` distinct, so
  a future 38.x.d could surface this difference in T802 (rejecting a
  `persist` value INTO a `GENERATED ALWAYS` column ‚Äî Postgres rejects
  it server-side, but a compile-time check is better). Today T803 is
  the only consumer and treats both the same (omittable).
- The `auto_increment` field is **NOT deprecated** ‚Äî adopters with
  legacy SERIAL still declare it. v1.38.3 makes axon's introspect emit
  it on SERIAL columns (unchanged from v1.38.2) AND `identity: true`
  on IDENTITY columns (new). A future v2.0 may unify under a single
  enum, but 38.x.c keeps the two-field surface to preserve
  adopter-facing readability of the manifest.

# ‚ñ∂ 6. What is intentionally NOT in v1.38.3

- **T802 rejection of values INTO `GENERATED ALWAYS` columns.** Postgres
  already rejects this server-side; a compile-time arm would be nice
  but deferred to a future 38.x.d. The `attidentity` distinction is
  preserved in `IntrospectionRow` so the next cycle can pick it up
  without changing the manifest format.
- **Backwards-compat aliasing `auto_increment` ‚Üí `identity`.** They
  stay distinct fields. An adopter hand-writing `"auto_increment": true`
  in a manifest for an IDENTITY column will still work (T803 skips it
  via the existing `auto_increment` branch), just not via the SQL-
  precise field.
- **Migration of pre-v1.38.3 introspected manifests** ‚Äî adopters who
  introspected against v1.38.2 have manifests that miss the
  `identity: true` flag on IDENTITY columns. They re-run `axon store
  introspect` against v1.38.3 to get the updated manifest. The OLD
  manifest still parses (D5) but T803 keeps firing until the manifest
  is refreshed.
