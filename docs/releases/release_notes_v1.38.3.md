## The kivi smoke-16 third finding

The adopter's DDL:

```sql
CREATE TABLE chat_history (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    tenant_id UUID NOT NULL,
    content TEXT NOT NULL,
    ...
);
```

The adopter's `persist`:

```axon
persist into chat_history {
    tenant_id: ${tid}
    content:   ${msg}
}
```

Pre-v1.38.3 error:

```
axon-T803 `persist` omits NOT-NULL column `id` (BigInt, primary_key)
          with no default; the column must be provided or have a
          default in the schema declaration.
```

The workaround the adopter had to apply: **DELETE `id` from the
manifest entirely**. That hides the column from the type-checker —
silencing T801 + T802 + T804 for every future reference. The whole
Pillar V TypedColumn promise undermined for one false-positive.

## Root cause: two-channel auto-fill, single-channel introspect

Postgres represents column auto-fill via TWO independent surfaces:

1. **`pg_attrdef.adbin`** — the textual `DEFAULT <expr>`. The legacy
   `SERIAL` keyword expands to `INTEGER NOT NULL DEFAULT nextval(seq)`,
   so the marker shows up here.
2. **`pg_attribute.attidentity`** — a single-char marker for the
   modern SQL-standard `GENERATED ... AS IDENTITY` syntax:
   - `'a'` = `GENERATED ALWAYS AS IDENTITY` (Postgres rejects
     user-supplied values; `OVERRIDING SYSTEM VALUE` required to
     override)
   - `'d'` = `GENERATED BY DEFAULT AS IDENTITY` (auto-fills when
     omitted; user-supplied values accepted)
   - empty / null = not an identity column

Pre-v1.38.3 axon read **only channel #1**. The
`detect_auto_increment(default_expression)` helper looked for
`nextval(` — perfect for SERIAL, blind to IDENTITY. The doc-comment
at `store_introspect.rs:112` even acknowledged the gap:

> *"A column declared `GENERATED ... AS IDENTITY` does NOT appear in
> `pg_attrdef` (Postgres handles identity separately)…"*

But the code beneath that comment never queried `attidentity`. v1.38.3
closes the loop.

## The contract — five D-letters

| D-letter | Guarantee |
|---|---|
| **D1** | `pg_attribute.attidentity` is a first-class introspection field. The deep query in `axon-rs/src/store/introspect_cli.rs` SELECTs `a.attidentity::text AS identity_kind`; `decode_introspection_row` maps 'a'/'d'/empty to `IntrospectionRow.identity_kind: Option<char>`. **Statically enforced** by §4 grep §-assertion |
| **D2** | `identity: bool` is a first-class manifest + AST field. `StoreColumn` / `ManifestColumn` / `DeclaredColumn` / `IRStoreColumn` all carry it. Manifest canonical JSON OMITS the key when false (preserves byte-identity to v1.38.2 for non-IDENTITY columns) |
| **D3** | T803 treats `identity: true` as safely omittable. `has_default = !default_value.is_empty() \|\| auto_increment \|\| identity`. The NOT-NULL identity column is now structurally exempt from the omission check |
| **D4** | `auto_increment` semantics unchanged. SERIAL (via `nextval` default) emits `auto_increment: true`; IDENTITY (via `attidentity`) emits `identity: true`. The two SQL surfaces stay DISTINCT in the manifest — preserving the GENERATED ALWAYS vs BY DEFAULT distinction for future T802 arms |
| **D5** | **Absolute backwards-compat.** A v1.38.2 manifest parses with `identity = false` for every column AND re-serializes WITHOUT introducing an `identity` key. v1.38.2 → v1.38.3 upgrade is pure-additive on the manifest format |

## What you'll see post-upgrade

- **Run `axon store introspect chat_history --connection $DATABASE_URL --output schemas/chat_history.json`** against a table with IDENTITY columns. The output now includes `"identity": true` on those columns automatically.
- **A `persist` that omits a NOT-NULL IDENTITY column compiles green.** No more T803 false positives.
- **Every other proof for the column keeps working.** Because the column is no longer deleted from the manifest, T801 (unknown column), T802 (type mismatch), and T804 (field typo) all fire correctly on future references.
- **Hand-written manifests** can declare `identity: true` for an IDENTITY column directly, in addition to the existing `auto_increment: true` for SERIAL.
- **Adopters who introspected against v1.38.2 should re-run `axon store introspect`** to get the updated manifest with `identity: true` populated. The old manifests still parse (D5) but T803 keeps firing on those columns until the manifest is refreshed.

## What is intentionally NOT in v1.38.3

- **T802 rejection of values INTO `GENERATED ALWAYS` columns.**
  Postgres already rejects this server-side; a compile-time arm would
  be nice but deferred to a future Fase 38.x.d. The `attidentity`
  distinction is preserved in `IntrospectionRow` so the next cycle can
  pick it up without changing the manifest format.
- **Source-level `identity` keyword on inline `schema { … }` blocks.**
  Today the inline form's parser does not recognize an `identity`
  keyword on columns — adopters who hand-write inline schemas with
  IDENTITY columns can use a manifest instead OR add `auto_increment`
  as a stopgap (T803 skips both). A future 38.x.d will add the
  parser surface.
- **Migration of pre-v1.38.3 introspected manifests.** Adopters who
  introspected against v1.38.2 need to re-run `axon store introspect`
  to get the updated manifest. The old manifest still parses but T803
  keeps firing.
